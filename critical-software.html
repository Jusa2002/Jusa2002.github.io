Critical Software

Software Tester · Hitachi Rail / Infraestruturas de Portugal · Lisboa

Context

This role doesn't follow a strict linear progression. While there are periods where certain tasks dominate, the reality is more fluid — depending on the team's needs at any given time, I rotate between manual testing, automated testing, and tool development. Some weeks are spent deep in a test campaign, others writing code to automate something that was taking too long by hand. That back-and-forth is actually what I enjoy most about it.

The project started at Thales — which became Hitachi Rail in June 2024 — and sits within the railway safety domain. The core product is interlocking software: the system that allows a command centre to route trains along a railway network while guaranteeing safety constraints are met — no collisions, no conflicting signals, no routing failures. It's safety-critical in the strictest sense.

What I do
Manual Testing
Since Oct 2023 · Ongoing

The entry point into the project — and a recurring responsibility throughout. Manual testing here means working with software control tables: large specification documents that define exactly how the interlocking software should behave under every possible scenario. For each test case, I verify that the software's actual output matches what the table dictates.

This phase required building deep domain familiarity before anything else — understanding track layouts, signal logic, route locking, and the safety rules that govern all of it.

Automated Test Parametrization
Since early 2024 · Recurring

After building enough domain knowledge, I moved into supporting the automated test infrastructure. Automated tests are driven by databases that need to be parametrized — filled with the right values to correctly represent each test scenario. Repetitive and time-consuming by nature, which planted the seed for the tooling work that followed.

⭐ Engineering highlight
Java Tool — Station as a Graph
2024

While filling parametrization databases, I kept running into the same repetitive pattern. At some point it clicked: a train station is fundamentally a graph. Track elements are nodes, routes are paths through that graph with defined start and stop conditions.

I built a Java tool that takes a CSV file encoding the structure of a train station and uses it to automatically populate the relevant sections of the parametrization database. What previously required significant manual effort could now be done with a small amount of structured input — consistently and without human error.

ETCS Validation — Radio Block Centre
2024 · Recurring

A separate stream of work involving the European Train Control System (ETCS). The task is to verify that the parametrization of the Radio Block Centre complies with the software's product rules — a more exhaustive set of checks than standard interlocking testing.

Given the volume of rules to verify, I adapted the tooling I had already developed to also support this validation work, rather than starting from scratch.

ETCS Radio Block Centre Compliance Validation
Concurrency Checker — XML State Machines
2025

Certain behaviours in the interlocking software are defined through state machines, parametrized via XML files. I developed a tool that parses those XML files and applies the necessary logic to verify concurrency conditions between states — catching configuration conflicts that would be difficult to spot manually.

XML Parsing State Machines Concurrency Analysis
XML Diff Tool
2025

A tool to parse and compare XML configuration files across versions, identifying and reporting differences in a structured way. Useful whenever a new software release changes the underlying configuration and the team needs to understand exactly what changed and where.
